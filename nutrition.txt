Build the Nutrition Wellness module

Goal:
Build a Nutrition Wellness module that helps users:
1. define protein and broader nutrition goals,
2. get meal suggestions based on ingredients they already have at home,
3. receive high-protein meal ideas tailored to their workout context and nutrition needs,
4. optimize authentic/traditional dishes to better match health or protein goals without completely losing the spirit of the dish,
5. save AI-generated recipes,
6. save their own modifications/tweaks to generated recipes,
7. add their own custom recipes manually,
8. view and manage saved recipes in a persistent recipe library.

Important:
This module is NOT a medical nutrition diagnosis system.
Do NOT present outputs as medical advice, treatment, or clinical nutrition prescription.
Frame results as meal-planning support, protein/nutrition guidance, recipe ideas, and wellness-oriented food suggestions.

========================
MODULE SCOPE
========================

Implement these sub-features:

1. Nutrition Goals / Profile Intake
- Create a form where the user can enter:
  - primary goal: muscle gain / fat loss / maintenance / general wellness / high protein
  - protein goal (grams/day optional)
  - calorie goal optional
  - dietary preferences: vegetarian / vegan / pescatarian / omnivore / halal / etc.
  - dietary restrictions or allergies
  - preferred cuisines
  - disliked ingredients
- Store this nutrition profile in the database.

2. Pantry / Ingredient-Based Meal Planner
- Let the user input ingredients they currently have at home.
- Use Gemini API to generate meal suggestions based on:
  - pantry ingredients,
  - nutrition goals,
  - dietary preferences/restrictions,
  - workout context if available.
- Workout-aware examples:
  - post-workout high-protein meal,
  - lighter recovery meal,
  - quick high-protein meal before class,
  - lower-calorie but filling option.
- Output should include:
  - recipe name
  - short description
  - estimated protein emphasis / goal fit
  - ingredients used
  - simple cooking steps
  - why it fits the user’s current goal

3. Authentic Dish Scraper + Protein Optimizer
- Build a feature where the user can search for a traditional/authentic dish or cuisine.
- Example user inputs:
  - “Pad Thai but more protein”
  - “Baingan bharta but healthier, less oil, extra protein”
  - “Authentic Indian dinner with more protein”
- The system should:
  - find authentic or traditional reference recipes from a scraper service or fallback recipe dataset,
  - preserve the spirit/core identity of the dish,
  - suggest modifications that improve protein content or overall nutrition fit,
  - optionally reduce oil, improve balance, or fit user goals without making the dish feel completely inauthentic.
- Output should clearly separate:
  - original/traditional baseline idea
  - suggested nutrition/protein-oriented modifications
  - explanation of what was changed and why

4. Save AI-Generated Recipes
- Let users save generated recipes to MongoDB.
- Saved recipes should store:
  - title
  - source type (generated / optimized / custom)
  - ingredients
  - steps
  - user nutrition goal context
  - tags
  - saved timestamp

5. User Tweaks / Recipe Modification
- Let the user edit or tweak an AI-generated recipe.
- Example:
  - add tofu
  - reduce oil
  - swap paneer for chicken
  - remove peanuts
  - increase serving size
- Save these user modifications separately or as versioned recipe edits.
- The system should preserve both:
  - original generated recipe
  - user-customized version

6. Add Custom Recipe
- Provide a separate feature where a user can manually add their own recipe from scratch.
- Fields:
  - recipe title
  - cuisine
  - ingredients
  - instructions
  - notes
  - optional nutrition intent tags
- Save to MongoDB.

7. Saved Recipe Library
- Build a recipe library where users can:
  - view saved AI-generated recipes,
  - view optimized authentic dish variants,
  - view their own custom recipes,
  - search/filter by tags, cuisine, protein goal, dietary type, or source type.

========================

========================

This feature should feel like:
- a practical nutrition assistant,
- a protein-focused meal planning companion,
- a culturally aware recipe optimizer,
- a recipe memory system that learns what the user likes.

It should NOT feel like:

- a generic AI chatbot with random meal ideas,
- a recipe blog clone.

The UI should feel:
- clean,
- warm,
- useful,
- easy to navigate,
- consistent with the larger NowWhat product.

========================
CORE LOGIC RULES
========================

1. Meal and recipe outputs should be practical and realistic.
Use language like:
- “high-protein option”
- “better fit for your current goal”
- “protein-forward version”
- “lighter variation”
- “more balanced version”
Do NOT use language like:
- “perfect diet”
- “medically required”
- “guaranteed fat loss”
- “this will cure”
- “this is the only healthy option”

2. Recipe suggestions must consider:
- user nutrition goals,
- dietary preferences/restrictions,
- pantry ingredients,
- saved profile,
- workout context if available.

3. Authentic dish optimization must preserve dish identity as much as possible.
Examples:
- suggest adding tofu, paneer, lentils, eggs, Greek yogurt, lean meat, edamame, or protein-rich sides where appropriate,
- reduce oil moderately when possible,
- do not completely transform the dish into something unrelated and still call it the same dish.

4. Keep core filtering and recipe data handling deterministic wherever possible.
Gemini should be used for:
- structured recipe generation,
- recipe optimization reasoning,
- explanation text,
- parsing flexible user input.
Do NOT rely on Gemini alone for:
- persistence logic,
- filtering saved recipes,
- source-of-truth storage,
- recipe library state.

5. For MVP, use a small local mock dataset for:
- authentic dish examples,
- recipe seeds,
- pantry-friendly meal templates,
if live scraping is not fully implemented yet.

========================
GROUNDING RULE
========================

Every recommendation, recipe suggestion, optimization output, or insight in this Nutrition Wellness module must include one concise grounding line.

Format:
Also check: [most relevant source or document]

This line must always be present.

Use context-appropriate grounding such as:
- official nutrition label
- recipe ingredient list
- trusted recipe/source page
- restaurant menu page
- evidence-based dietary guideline
- product/package instructions when relevant

Examples:
- Also check: the recipe ingredient list and serving sizes before cooking.
- Also check: the official nutrition label for exact protein and calorie details.
- Also check: a trusted recipe source for the traditional baseline version of this dish.
- Also check: the restaurant’s official menu or nutrition page for exact details.

Do not skip the grounding line.
Do not phrase it like a legal disclaimer.
Keep it natural and concise.

========================
GEMINI API USAGE
========================

Use Gemini API as the primary generation and reasoning layer for:
- pantry-based meal suggestions,
- nutrition-goal-aware recipe generation,
- authentic dish optimization,
- recipe explanation text,
- user tweak interpretation.

Gemini outputs should be structured and machine-readable wherever possible.

Preferred Gemini use cases:
- generate recipes from ingredient lists and goal context,
- suggest protein-oriented or lighter modifications to traditional dishes,
- explain why a recipe fits a user’s goals,
- interpret user tweak requests like:
  - “make it less oily”
  - “add more protein”
  - “make it vegetarian”
  - “keep it authentic but healthier”

Gemini must NOT:
- make medical claims,
- act as a dietitian diagnosis engine,
- invent false nutrition certainty,
- override deterministic storage and filtering logic.

If Gemini is temporarily unavailable, support fallback/mock recipe generation through local templates or seeded recipe data.

========================
SCRAPING / AUTHENTIC RECIPE RULES
========================

For the authentic dish feature:
- support a scraping service or recipe-source abstraction layer,
- if live scraping is unavailable for MVP, use a fallback local dataset of authentic/traditional dish examples,
- keep the architecture ready for future real scraping integration.

When presenting optimized authentic dishes:
- show the traditional baseline concept,
- show the optimized version,
- explain what changed,
- avoid pretending the optimized version is the only “correct” form of the dish.

========================
OUTPUT STYLE REQUIREMENTS
========================

Whenever the module generates a result for the user, structure it as:

1. Main result
2. Why it fits / what changed
3. Confidence level if relevant
4. Also check: ...

Example:
Result: This is a higher-protein version of Pad Thai that keeps the core sweet-salty-tangy flavor profile while improving protein balance.
Why it fits: The recipe adds tofu/egg/chicken options, moderates oil slightly, and keeps the core sauce and noodle structure recognizable.
Confidence: Medium
Also check: a trusted Pad Thai recipe source and the ingredient labels for exact protein values.

Do NOT overbuild:
- no medical nutrition claims,
- no advanced macro-accuracy promises unless supported by real nutrition data,
- no giant recipe knowledge graph,
- no overly complex scraping pipeline for MVP.
