Build the Skin & Hair Health module 
Goal:

Update the Skin & Hair Health service that helps users:

1. define their skin and scalp profile,

2. upload skin and/or scalp photos for broad visual analysis,

3. save products they already love,

4. get personalized product recommendations based on ingredient similarity and sensitivity filtering,

5. see lightweight wellness correlations between skin/hair changes and lifestyle signals like sleep, workout fatigue, nutrition, and supplements.
 
Important:

This module is NOT a medical or dermatology diagnosis system.


Do NOT present outputs as diagnosis, treatment, or clinical certainty.

Frame results as visible pattern estimation, product-fit guidance, and wellness-pattern observations.
 
========================

MODULE SCOPE

========================
 
Implement these sub-features:
 
1. Skin & Scalp Profile Intake

- Create a form where the user can enter:

  - skin type: dry / oily / combination / sensitive

  - scalp type: dry / oily / balanced / sensitive

  - concerns: acne, dryness, oiliness, dark circles, dandruff, thinning, irritation

  - allergies

  - ingredient sensitivities (example: fragrance, sulfates, parabens)

- Store profile data in the database.
 
2. Skin Image Analysis

- Allow upload of a face photo.

- Use Gemini Vision as the primary MVP image-analysis layer.

- Return broad, non-medical appearance scores such as:

  - dryness_score

  - oiliness_score

  - acne_like_appearance_score

  - dark_circles_score

- Keep this broad and cautious.

- If Gemini integration is temporarily unavailable during development, use placeholder/mock scoring logic behind the same analysis service interface so it can be replaced cleanly later.
 
3. Hair / Scalp Image Analysis

- Allow upload of a scalp or hairline photo.

- Use Gemini Vision as the primary MVP image-analysis layer.

- Return broad, non-medical appearance scores such as:

  - scalp_dryness_score

  - dandruff_like_flaking_score

  - thinning_appearance_score

- Keep output cautious and non-diagnostic.

- If Gemini integration is temporarily unavailable during development, use placeholder/mock scoring logic behind the same analysis service interface so it can be replaced cleanly later.
 
4. Loved Product Entry

- Let users save products they already like and use successfully.

- Fields:

  - product name

  - brand

  - category (cleanser, moisturizer, serum, shampoo, conditioner, scalp serum, etc.)

  - ingredients

  - optional notes

- Support manual ingredient entry if no external ingredient parser exists yet.
 
5. Ingredient Fingerprint + Product Recommendation

- Build logic that:

  - identifies ingredient overlap across loved products,

  - creates a simple positive ingredient fingerprint,

  - filters out ingredients listed in allergies/sensitivities,

  - recommends products from a product dataset or mock dataset.

- Recommendations should include:

  - product name

  - category

  - match score

  - matched ingredients

  - avoided blocked ingredients

  - short reason
 
6. Wellness Correlation Cards

- Read lightweight lifestyle signals from existing or mock wellness logs:

  - sleep duration / quality

  - workout fatigue score

  - nutrition quality or protein consistency

  - supplement consistency

- Show simple pattern statements such as:

  - “Dryness scores appear higher on low-sleep days.”

  - “Scalp dryness appears elevated during weeks with inconsistent nutrition.”

- These are observational patterns only, not causal or medical claims.

CORE LOGIC RULES

========================
 
1. Analysis outputs must be broad and cautious.

Use wording like:

- “possible dryness signal”

- “visible flaking pattern”

- “appearance suggests”

- “may align with”

Do NOT use wording like:

- “you have”

- “you are diagnosed with”

- “this proves”

- “this is caused by”
 
2. Product recommendations must be based on:

- ingredient overlap with loved products,

- profile compatibility,

- avoidance of allergens/sensitivities,

- category relevance.
 
3. Wellness insight generation must be observational only.

Examples:

- “Dryness appears higher on low-sleep days.”

- “Higher fatigue weeks align with elevated under-eye score.”

Do NOT claim medical causation.
 
4. Keep the recommendation logic deterministic wherever possible.

Use simple scoring formulas for MVP.

LLM use is optional for explanation text, but do not rely on an LLM for core filtering logic.
 
For MVP, use a small local mock dataset (JSON or seeded database collection) of skincare and haircare products with category and ingredient lists. Gemini is allowed for image-analysis extraction and optional explanation text.

Deterministic logic must be used for product filtering, sensitivity exclusion, recommendation scoring, and wellness correlation calculations.
 
========================

GROUNDING RULE

========================
 
Every recommendation, insight, or analysis result in this Skin & Hair module must include one concise grounding line.
 
Format:

Also check: [most relevant source or document]
 
This line must always be present.
 
Use context-appropriate grounding such as:

- product ingredient list

- dermatologist guidance

- brand product page

- patch test instructions

- official usage instructions

- clinician-approved resource when relevant
 
Examples:

- Also check: the full ingredient list on the product label to confirm it avoids your sensitivities.

- Also check: the brand’s official product page for directions and ingredient details.

- Also check: a dermatologist-approved skincare resource for guidance on persistent irritation.

- Also check: the shampoo or scalp product label for sulfate and fragrance content.
 
Do not skip the grounding line.

Do not phrase it like a legal disclaimer.

Keep it natural and concise.

========================

IMAGE PRIVACY & STORAGE RULE

========================
 
Do NOT permanently store user-uploaded face, skin, scalp, or hair images.
 
Image handling rules:

- Use uploaded images only for immediate analysis.

- After analysis completes, discard the raw image file.

- Do not save the original image to persistent storage, object storage, or database.

- Do not retain image URLs for uploaded user images.

- Do not use uploaded images for model training.

- Do not reuse uploaded images across sessions.
 
Instead, store only:

- analysis outputs/scores

- timestamps

- user profile data

- concerns

- wellness insight summaries

- product recommendation history
 
If temporary file storage is required during processing:

- keep it short-lived,

- delete immediately after analysis,

- avoid exposing file paths publicly.
 
The goal is to preserve user privacy while still building longitudinal memory from structured analysis results rather than raw photos.
 
For each uploaded image, persist only derived metadata such as:

- dryness_score

- oiliness_score

- acne_like_appearance_score

- dark_circles_score

- scalp_dryness_score

- dandruff_like_flaking_score

- thinning_appearance_score

- analysis_timestamp

- confidence level

- related concerns
 
Do not persist:

- raw image bytes

- image file path

- public image URL

- base64 image content

Important rules:

- Do NOT use Gemini for diagnosis.

- Do NOT allow medical claims, treatment suggestions, or disease naming.

- Keep all outputs limited to visible appearance estimation and wellness-pattern observation.
 
Allowed phrasing:

- possible dryness signal

- mild under-eye fatigue appearance

- visible flaking pattern

- possible thinning appearance signal
 
Forbidden phrasing:

- you have acne

- you have dandruff

- you have a deficiency

- this is caused by

- diagnosis or treatment language
 
After Gemini returns structured output, apply a post-processing guardrail layer that:

- validates fields,

- rewrites unsafe wording into safe appearance-based language,

- preserves only supported structured outputs,

- adds the required grounding line.

Use images only for immediate Gemini analysis, then discard them.

Store only:

- structured analysis results,

- timestamps,

- confidence,

- concerns,

Gemini responses for image analysis must be schema-constrained and machine-readable.

If Gemini returns unsupported, overly certain, or medicalized language, discard that phrasing and replace it with safe appearance-based wording before saving or displaying results.
